1.1.1 c,c++ 编译器

csdn 参考

  • 主流的就是那三个：Visual C++、gcc、Intel C/C++ Compiler。
  • 编译器编译出的代码，性能最强大的，毫无疑问的，intel
    • 要说库，intel mkl，ipp，都是一骑绝尘的水平。尤其是 fft 之类的
    • 对于 openMP 这类 工程科研人员使用的并行指令。VS 好像仅支持 2.0？版本。Intel 的一向支持最新版本。
    • 对于 SSE 等 CPU 指令集，VS2010 仅仅到 SSE2。intel 的编译器，都已经 avx 了
    • 对于 profile。VS 的性能测试，几乎没用. Intel 的， 可以强大到具体针对某款 CPU 的 μops
  • 用 intel parallel studio，
    • 无论是 编译器性能，parallel debug，security checker（memory，threads，resource 泄露，竞争等），vtune 更是一骑绝尘的性能调优
    • 还有数一数二的数学库，mkl，ipp，
    • 或者，搞超算，搞 HPC，可以用 PGI 编译器。
    • VC 连 C 99 都不完整支持，
    • 科研常用的 openMP, vc 依旧只支持古老的 openMP 2.0
  • 产出代码的性能上 gcc,icc,vc 都是第一阶梯的.但各有各的优势.
    • gcc 的优势在于多平台,开源,以及一整套的开源工具.
    • icc 的优势在于它那套高性能库,及开发工具.不过总觉得它有些方面做的不好,比如:资源泄漏冲突检查方面,以前的 purify,boundcheck,甚至 AQTime 都要比它做的强些.
    • vc 的最大优势就在于优秀的 IDE 以及 MSDN library 了.但 MSDN library….时好时坏,一方面的改善就伴随着另一方面极大的退步.
    • 单从编译器来看.他们并没有太大的差距… 就我使用的感觉来看.
      • 一般情况下最新的 gcc 4.7 系列,生成的代码质量稍高一点.但编译速度稍慢.
      • vc2010 sp1 比较稳定,没有特别突出的地方.vc2012 在一些方面有改善,但目前 bug 比较严重.已经多次遇到优化代码出错的问题了.编译速度快.
      • icc 系列在单纯的数学运算,并行化/矢量化方面有明显优势.但对于其它类型程序,它经常比上两者要差点.最要命的是它编译速度太慢…
  • 感受
    • 1、界面以及辅助部分（数据库、网络、UI）用 Qt，嵌入到 VC 里去做。
    • 2、尽量使用现成的并行化工具（OpenMP、Qt 的 Concurrent)，避免自己写多线程。学院派的大拿会被多线程调试玩死。
    • 3、算法部分，除非非常重视性能，不要使用与界面部分不同的编译器，那样调试起来很麻烦，出了问题会踢皮球。
    • 4、少用 COM，这个东西太复杂，博士大人们要不叶公好龙，要不嗤之以鼻。就用标准的 C 接口最稳定，写好文档就 OK。如果需要复杂的接口，就用 Qt 的 plugin 体系，做的很快，很强大。

1.1.3 DONE building system

参考 zhihu1

用其它更高级的工具根据平台，额外配置来生成 makefile（或者生成常用 IDE 的工程文件）, 之后再调用 make。cmake 就是这样的工具，类似的工具还有 Qt 用的 qmake，Lua 写的 premake。

作为一个专注 c++工程的 DSL，cmake 具有以下优势：
1 跨平台支持，可生成 vs, makefile, xcode, eclipse 等工程，且有诸多 ide 支持
2 支持依赖管理，包括 include dirs, lib path, compile flags 等 c++工程特有属性
3 通过 custom toolchain file 可以支持很多编译工具链，方便交叉编译
4 较成熟，相对 bug 少
5 手写 findxxx.cmake，容易扩展
6 带部分 distribution 辅助功能(INSTALL & PACK)
7 test 支持

make 和 cmake 是两个层级，make 主要解决增量编译的问题，cmake 主要则是分析依赖和跨平台对接 IDE. 一个项目的依赖其实是很复杂的树形结构
