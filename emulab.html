3.1.7.2 XEN support

> 0) first: our local installation just isn't running Xen
> virtualization. I'm debugging, and climbing the learning curve
> simultaneously. I'll have details and pointed questions later; but so
> far I'm studying the failure.

If the experiment is failing to map, a good thing to try is:

boss> mapper -d -n -v $pid $eid

which will provide lots of output that will help us debug the problem.

> 1) there appears to be a hard-coded router IP address for the virtual
> nodes' control subnet. Our defs file specifies the default
> 172.16.0.0/12, but there's no config variable or documentation on how
> to route between the VIRTNODE subnet and the physical node control
> subnet. The vhost scripts apparently assume the router is 172.16.0.1,
> and set up the firewall, routing tables, and bridging accordingly. Is
> there a missing config variable?

Buried in the install documentation are a few sentences about having to
setup your router to route between the 172 network and your emulab networks.

Note that you cannot change the network from 172.16 … even though its a
config variable, don't change it!

> 2) it would be convenient to have physcial and virtual nodes allocated IP
> addresses from the same subnet. The sample defs file comment says to
> contact Utah before changing the defaults. Would I be able to set the
> two subnets to be identical?

No, but you can add parts of your routable network that will be used for
control network addresses of the vhosts.

Usage: addvpubaddr [-d] aaa.bbb.ccc.ddd eee.fff.ggg.hhh [mmm.nnn.ooo.ppp]
where aaa.bbb.ccc.ddd is the first addresses to add,
eee.fff.ggg.hhh is the last address to add,
mmm.nnn.ooo.ppp is the optional netmask to apply
(default is control network netmask).
if -d is specified, the addresses in range are deleted.

Gary can say more about this; I cannot find the documentation in the wiki.

> 3) running the sample Xen ns script from https://wiki.emulab.net/wiki/xen
> (and minor variations) gives inconsistent results. Using the
> 'pcvm3000' hardware type causes failures. None of the standard VM OS
> candidates appear to boot successfully except the default
> (XEN-STD). The only combination that seems to work (intermittently) is
> hardware: pcvm, OS: XEN-STD. I've seen this succeed before. But I just
> ran it again, allocating pc206 as host, and the Xen VMs fail to
> boot. I'll leave it swapped in.

Sorry, there is a problem with the XEN image on our pc3000s which have only
2GB of memory. Add this to your NS file:

$node add-attribute XEN_MEMSIZE 512

for each of your three nodes. I hope to have this fixed this week.
>
> 4) an observation: the /etc/hosts file on the vm host has IP addresses
> for the experiment interfaces, which are unreachable from the host
> – making them useless.

You need to turn on routing in your NS file. See the tutorial or the FAQ.

> 5) a question: I don't understand why there are two methods for accessing
> Xen VMs: a) an IP address allocated for the cnet interface for each
> VM, and b) traffic routed through the host's IP over an assigned
> port. Either should be sufficient, if consistently used? SNAT/DNAT
> mangling seems like the more complex approach.

Because if you want to reach a VM using an unroutable address from outside
your control network, you have to use the routable cnet address of the
physical host, and the port takes you inside the VM you want.

On the local network though, it is handy to ssh to pcvmxxx-yyyy, but that
does not work from outside the network.

3.2 安装 seer

3.2.1 基础

3.2.1.1 查找问题的办法
  1. 阅读 wiki, 调试源码
  2. 查看日志, 由源码知默认主日志是 /local/logs/setup.log, 同目录下还会有相关日志
  3. 抓包
3.2.1.2 机制
  • 主要组件是 Agent, Service, Collector, Aggregator
  • messaging network 由两部分组成. 一是 control 到实验节点 multicast & SSL socket,
    二是 control 到 gui 或 script via SSL sockets
3.2.1.3 帐号

3.2.2 步骤

  1. ./deploy /share/seer

  2. 在创建实验时,ns 脚本中加入:

    tb-set-node-startcmd $node "sudo python /share/seer/experiment-setup.py Basic"

  3. 实验 ns 脚本中加入 control 节点.

    在使用 SEER 进行实验,需要在通常的实验节点之外,新建一个 control 节点,
    control 节点不与其它节点连接, 也不参与实验, 主要负责监测网络流量
    等实验事件并与 ops 服务器联系。

  4. seer gui 连接

    用户从 SEER GUI 从 Emulab 接口处,输入项目、实验名与控制节点,以及
    Emulab 用户名、主机名(ops 的 URL),openSSH 私钥文件即可登录 SEER GUI 客户端

Hi. In red-dot mode, click on the Experimentation dropdown, then the
Image List. Find that image and click, then use the Edit link. You
need to check the hardware types that the image is able to run on.
Then submit.

You might need to do this for other images as well.

The likely problem is that the specified OS image is not marked as
compatible with your node type(s). Go to Experimentation->List
ImageIDs, find the image in question and click on it, then edit its
settings by clicking "Edit this Image Descriptor" in the left-hand
menu. Check the box next to each node type you believe the image
should work on, then click "submit" to save. Try firing the image up
on each of the node types to confirm that it loads and boots up.
